## 1. 完整认证链路追踪

### 1.1 认证流程图（文字描述）

```
┌──────────────────────────────────────────────────────────────────────┐
│                        GrowthMind 认证链路                            │
└──────────────────────────────────────────────────────────────────────┘

【登录阶段】
  用户输入邮箱密码
        │
        ▼
  login/page.tsx (L65-66)
  supabase.auth.signInWithPassword()
        │
        ▼
  Supabase Auth 返回 session.access_token
        │
        ▼
  document.cookie = "sb-access-token=<JWT>; path=/; max-age=3600; SameSite=Lax"
  (login/page.tsx L76, auth-provider.tsx L46)
        │
        ▼
  AuthProvider 监听 onAuthStateChange → 同步 cookie + 用户状态
  (auth-provider.tsx L136-156)

【页面访问阶段】
  浏览器请求页面
        │
        ▼
  middleware.ts (L40-42)
  读取 cookie: sb-access-token / sb-growthmind-auth-token
  或读取 Authorization: Bearer <token>
        │
        ├─ 无 token 且非公开路由 → 重定向 /login (L48-56)
        ├─ 管理员路由 → 验证 token + 查询 profile.role (L59-94)
        └─ 普通路由 → 仅检查 token 存在，不验证有效性 ❌

【API 调用阶段】
  前端 useFetch/useMutation (use-api.ts L36, L86)
        │
        ▼
  添加请求头: x-session: <access_token>
  （注意：不是 Authorization，也不是 Cookie）
        │
        ▼
  middleware.ts → 检查 Cookie 存在（浏览器自动携带）→ 放行
        │
        ▼
  api-auth.ts authenticateRequest() (L13-26)
  读取 x-session 请求头 → supabase.auth.getUser(token) 验证
        │
        ├─ 验证失败 → 401
        └─ 验证成功 → 返回 { user, token }

【数据查询阶段】
  API 路由调用 db (drizzle-orm) 查询数据库
        │
        ▼
  db 使用 DATABASE_URL 直连 PostgreSQL (lib/db.ts)
  ❌ 不经过 Supabase，不携带用户 JWT，RLS 不生效
  仅靠应用层 WHERE userId = auth.user.id 过滤
```

### 1.2 认证链路断点分析

| 断点编号 | 位置 | 问题描述 | 风险等级 |
|---------|------|---------|---------|
| 断点-1 | `middleware.ts` L40-56 | 中间件仅检查 cookie/header **是否存在**，不验证 token 有效性（非管理员路由）。伪造 cookie 值即可绕过中间件 | 高 |
| 断点-2 | `api-auth.ts` L13 vs `middleware.ts` L40-42 | 中间件读取 `cookie` 或 `Authorization` 头，API 认证读取 `x-session` 头。**两套认证机制不一致**，中间件放行的请求 API 层可能拒绝，反之亦然 | 高 |
| 断点-3 | `profile/route.ts` L10-31 | `/api/profile` **完全未调用 `authenticateRequest`**，仅靠 `userId` 查询参数获取用户信息，任何人可查询任意用户资料 | **严重** |
| 断点-4 | `middleware.ts` L20 | `/api/supabase-config` 被列为公开路由，该接口返回 Supabase URL 和 anonKey，虽属设计意图，但 anonKey 暴露到前端 | 中 |
| 断点-5 | `lib/db.ts` L1-17 | 数据库查询使用 drizzle-orm 直连 PostgreSQL（`DATABASE_URL`），**不经过 Supabase Auth**，RLS 策略完全不生效 | **严重** |

### 1.3 Cookie 名称一致性

| 设置位置 | Cookie 名称 | 文件行号 |
|---------|------------|---------|
| 登录页 | `sb-access-token` | `login/page.tsx` L61, L76 |
| AuthProvider | `sb-access-token` | `auth-provider.tsx` L46, L48 |
| 中间件读取 | `sb-access-token` + `sb-growthmind-auth-token`（兼容） | `middleware.ts` L40-41 |
| 数据层读取 | `sb-access-token` + `sb-growthmind-auth-token`（兼容） | `lib/data/goals.ts` L11-12, `lib/data/records.ts` L11-12 |

**分析**：`sb-access-token` 在设置和读取侧一致。但 `sb-growthmind-auth-token` 在代码中**从未被设置**，仅在读取侧作为兼容兜底，属于无效的防御代码。

### 1.4 Token 格式和验证方式

- **Token 格式**：Supabase JWT access_token（标准 JWT）
- **验证方式**：
  - 中间件（管理员路由）：`supabase.auth.getUser(accessToken)` 验证 ✓
  - API 路由：`supabase.auth.getUser(token)` 验证 ✓
  - 中间件（普通路由）：**不验证** ❌
  - 数据层（`lib/data/*.ts`）：`supabase.auth.getUser(accessToken)` 验证 ✓

### 1.5 退出登录状态清理

| 清理项 | 位置 | 状态 |
|-------|------|------|
| Supabase 会话 | `auth-provider.tsx` L174 `supabase.auth.signOut()` | ✅ |
| React 状态 (user/session/profile) | `auth-provider.tsx` L175-177 | ✅ |
| Cookie `sb-access-token` | `auth-provider.tsx` L178 `setAuthCookie(null)` → `max-age=0` | ✅ |
| Redis 缓存 | 未清理 | ⚠️ 退出后缓存仍残留，但缓存键含 userId，不会泄露给其他用户 |

### 1.6 风险总结

**风险等级：高**

认证链路存在严重断点：
1. `/api/profile` 完全无认证，可越权查询任意用户资料（**最严重**）
2. 中间件与 API 层使用不同的认证机制（cookie vs x-session），链路割裂
3. 中间件对普通路由不验证 token 有效性，仅检查存在性
4. 数据库直连绕过 RLS，所有数据安全仅依赖应用层 WHERE 条件

## 2. Cookie 安全属性检查

### 2.1 所有 Cookie 设置点

| # | 文件路径 | 行号 | Cookie 名称 | httpOnly | secure | sameSite | max-age | path |
|---|---------|------|------------|----------|--------|----------|---------|------|
| 1 | `src/app/(auth)/login/page.tsx` | L61 | `sb-access-token` | ❌ 未设置 | ❌ 未设置 | Lax | 3600s | `/` |
| 2 | `src/app/(auth)/login/page.tsx` | L76 | `sb-access-token` | ❌ 未设置 | ❌ 未设置 | Lax | 3600s | `/` |
| 3 | `src/components/auth/auth-provider.tsx` | L46 | `sb-access-token` | ❌ 未设置 | ❌ 未设置 | Lax | 3600s | `/` |
| 4 | `src/components/auth/auth-provider.tsx` | L48 | `sb-access-token` | ❌ 未设置 | ❌ 未设置 | - | 0 (清除) | `/` |
| 5 | `src/components/ui/sidebar.tsx` | L86 | `sidebar_state` | ❌ 未设置 | ❌ 未设置 | - | 1年 | `/` |

### 2.2 问题分析

#### 2.2.1 httpOnly 缺失（高风险）

所有认证 Cookie 通过 `document.cookie` 设置（`login/page.tsx` L61/L76, `auth-provider.tsx` L46），**无法设置 `httpOnly` 属性**。

**影响**：
- JWT access_token 可被任意 JavaScript 读取
- 一旦发生 XSS 漏洞，攻击者可直接窃取 token
- 违反 OWASP 认证 Cookie 安全规范

**根因**：认证 Cookie 应由服务端通过 `cookies().set()` 设置（Next.js Server Components/API Routes），而非客户端 `document.cookie`。

#### 2.2.2 Secure 属性缺失（高风险）

所有 Cookie 均未设置 `Secure` 属性。

**影响**：
- 在 HTTP 环境下 Cookie 会以明文传输
- 中间人攻击可截获 access_token

#### 2.2.3 SameSite 属性

认证 Cookie 设置了 `SameSite=Lax`，可防御大部分 CSRF 攻击。但 `sidebar_state` Cookie 未设置 `SameSite`，风险较低（非敏感数据）。

#### 2.2.4 过期时间

`max-age=3600`（1 小时）与 Supabase JWT 默认有效期一致，但存在以下问题：
- Cookie 过期后不会自动续期（虽 Supabase 客户端会自动刷新 token，但 Cookie 不会同步更新）
- 无 `refresh_token` Cookie，刷新后的 token 仅更新 Supabase 内存状态，Cookie 仍为旧值

#### 2.2.5 Path 属性

所有 Cookie 的 `path=/` 设置正确，覆盖全站。

### 2.3 风险总结

**风险等级：高**

| 检查项 | 状态 | 说明 |
|-------|------|------|
| httpOnly | ❌ 不符合 | 认证 Cookie 可被 JS 读取，XSS 风险极高 |
| Secure | ❌ 不符合 | 未限制 HTTPS 传输 |
| SameSite | ⚠️ 部分符合 | 认证 Cookie 为 Lax，非敏感 Cookie 未设置 |
| 过期时间 | ⚠️ 部分符合 | 1 小时合理，但刷新机制不完善 |
| Path | ✅ 符合 | 统一为 `/` |

### 3.2 未启用 RLS 的表（风险点）

#### 3.2.1 `analysis_history`（高风险）

- **用途**：存储用户 AI 分析历史记录，含 `userId`、`timeRange`、`dimensions`、`result`

#### 3.2.2 `gateway_usage_log`（高风险）

- **用途**：记录 AI 网关调用日志，含 `userId`、`provider`、`model`、`tokensIn`、`tokensOut`、`cost`
- **风险**：暴露用户使用量、成本等敏感商业数据
- **当前应用层防护**：代码中未发现对该表的查询操作（仅 schema 定义）

#### 3.2.3 `email_templates`（高风险）

- **用途**：邮件模板，含 `subject`、`body`、`createdBy`
- **风险**：任何人可读取/篡改邮件模板内容
- **当前应用层防护**：代码中未发现对该表的查询操作

#### 3.2.4 `health_check`（低风险）

- **用途**：健康检查表，仅含 `id` 和 `updatedAt`
- **风险**：信息泄露极小

### 3.3 策略操作覆盖缺口

#### 3.3.1 `profiles` 表缺少 DELETE 策略

```sql
-- 0001_rls_policies.sql L44-52
-- 仅有 SELECT / INSERT / UPDATE，无 DELETE
```

**影响**：普通用户无法删除自己的 profile（RLS 会拒绝），仅管理员可通过 `FOR ALL` 策略删除。若产品需要用户注销功能，将受阻。

#### 3.3.2 `supervision_relations` 表缺少 DELETE 策略

```sql
-- 0001_rls_policies.sql L80-90
-- 仅有 SELECT / INSERT / UPDATE，无 DELETE
```

**影响**：代码中采用软删除（`active: false`，见 `supervise/[id]/route.ts` L19-24），所以应用层不依赖 DELETE。但若直接通过数据库操作，无法物理删除。

### 3.4 策略逻辑正确性

#### 3.4.1 用户数据隔离策略（正确）

```sql
-- records 表示例
CREATE POLICY "Users can view own records" ON records
  FOR SELECT USING (auth.uid() = user_id);
```

所有用户策略均使用 `auth.uid() = user_id` 进行隔离，逻辑正确。

#### 3.4.2 监督关系双方可见策略（正确）

```sql
-- 0001_rls_policies.sql L81-84
CREATE POLICY "Users can view own supervision relations" ON supervision_relations
  FOR SELECT USING (
    auth.uid() = admin_user_id OR auth.uid() = supervised_user_id
  );
```

管理员和被监督用户均可查看，符合业务需求。

#### 3.4.3 管理员策略（存在递归风险）

```sql
-- 0001_rls_policies.sql L108-111
CREATE POLICY "Admins can manage all profiles" ON profiles
  FOR ALL USING (
    EXISTS (SELECT 1 FROM profiles WHERE user_id = auth.uid() AND role = 'admin')
  );
```

**分析**：管理员策略在 `profiles` 表上查询 `profiles` 表本身，存在递归查询风险。PostgreSQL 通常通过 `auth.uid()` 函数避免无限递归，但建议改用 `auth.jwt() ->> 'role'` 直接从 JWT 提取角色，避免子查询。

### 3.5 RLS 实际生效问题（严重）

**核心问题**：应用层数据查询使用 `drizzle-orm` 直连 PostgreSQL（`lib/db.ts`），连接字符串为 `DATABASE_URL`，该连接使用的数据库角色（通常为 `postgres` 超级用户或 `service_role`）**绕过 RLS**。

```typescript
// lib/db.ts L6-17
const connectionString = process.env.DATABASE_URL;
const pool = new Pool({ connectionString, ... });
export const db = drizzle(pool, { schema: { ...schema, ...relations } });
```

**影响**：即使 RLS 策略定义正确，**所有通过 `db` 对象执行的查询均不受 RLS 保护**。RLS 仅在通过 Supabase 客户端携带用户 JWT 时生效，而本项目的数据查询路径完全不经过 Supabase。

**当前防护**：完全依赖应用层 `where(eq(xxx.userId, auth.user.id))` 过滤条件。

### 8.3 总体评价

GrowthMind 项目在**敏感信息管理**（环境变量、密钥存储）和**SQL 注入防护**方面表现良好，drizzle-orm 的参数化查询和 Next.js 默认同源策略提供了基础安全保障。

但在**认证链路完整性**、**Cookie 安全**和**RLS 实际生效**三个核心安全维度存在严重缺陷：

1. **认证链路割裂**：中间件（cookie）、API 层（x-session 头）、数据层（cookie）使用三套不同的认证机制，且 `/api/profile` 完全无认证。
2. **Cookie 安全不达标**：认证 token 通过 `document.cookie` 设置，无法享受 `httpOnly` 和 `Secure` 保护，XSS 攻击面极大。
3. **RLS 形同虚设**：虽然定义了 RLS 策略，但 drizzle 直连 PostgreSQL 使用超级用户角色，RLS 完全不生效，数据安全仅靠应用层 WHERE 条件兜底。

**建议在修复 P0 级漏洞前，不要将系统部署到生产环境。**
