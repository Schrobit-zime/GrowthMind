## 2. 转圈根因分析

### 2.1 根因 #1（直接原因）：`useFetch` loading 死锁 【P0】

**文件**：`src/hooks/use-api.ts` 第 28-29 行

**问题代码**：
```typescript
const [state, setState] = useState<FetchState<T>>({
  data: null,
  loading: enabled,  // ← 初始 true
  error: null,
});

const fetchData = useCallback(async () => {
  if (!session?.access_token || !enabled) return;  // ← 直接 return，不设 loading=false
  // ...
}, [url, session, enabled]);
```

**机制**：
1. `loading` 初始值为 `enabled`（默认 `true`）
2. 当 `session?.access_token` 为 `null` 时，`fetchData` 直接 `return`
3. `loading` 永远不会被设置为 `false`
4. 页面 `if (isLoading) return <LoadingSkeleton />` 永远成立 → **无限转圈**

**触发条件**（任一即可）：
- 未登录用户进入主页面
- 已登录但 config 加载失败 → session=null
- 已登录但 getSession 超时 → 5s 后 isLoading=false 但 session=null

**同样的问题存在于**：
- `src/app/(main)/records/page.tsx` 第 49-53 行（手动管理 loading，同样模式）
- `src/app/(main)/page.tsx` 主页（useFetch 未传 `enabled`，默认 `true`）

### 2.2 根因 #2（触发使能者）：缺少客户端认证守卫 【P0】

**文件**：`src/app/(main)/layout.tsx`

**问题**：
- 只渲染 `<AppLayout>{children}</AppLayout>`，**无任何认证守卫**
- 注释写"认证检查已由 middleware 统一处理"
- `AppLayout` 组件也不检查 `user`/`isLoading`，不重定向到 `/login`

**机制**：未登录用户访问 `/` → proxy 放行（/login 是公开路由，但 / 也需要 token）→ 进入 `HomePage` → `session=null` → 触发根因 #1 → 无限转圈

### 2.3 根因 #3（加剧因素）：`setAuthCookie` 未 await 【P1】

**文件**：`src/components/auth/auth-provider.tsx` 第 112 行

**问题代码**：
```typescript
if (currentSession?.user) {
  setSession(currentSession);
  setUser(currentSession.user);
  setAuthCookie(currentSession.access_token);  // ← 未 await，fire-and-forget
  await fetchProfile(currentSession.user.id);
}
```

**机制**：`setAuthCookie` 内部调用 `fetch("/api/auth/set-cookie")`，未 await。如果此请求慢或失败，cookie 可能未设置，但代码继续执行。后续请求缺少 cookie → proxy 重定向 → 循环。

### 2.4 根因 #4（API 层）：流式响应无超时保护 【P0】

**文件**：
- `src/app/api/analysis/route.ts` 第 74 行 fetch + 第 103 行 `while(true)`
- `src/app/api/gateway/route.ts` 第 123 行 fetch + 第 147 行 `while(true)`

**问题**：两处 `fetch` 调用外部 AI 服务（deepseek/openai/claude）均未设置 `signal: AbortSignal.timeout(...)`。`while(true)` 循环读取 SSE 流，无空闲超时检测。AI 服务 hang 时前端永久 await。

### 2.5 根因 #5（配置层）：`loadEnv()` 同步阻塞 【P1】

**文件**：`src/storage/database/supabase-client.ts` 第 34 行

**问题**：`execSync` 调用 python3，首次调用最多阻塞 Node.js 主线程 10 秒。期间所有请求被阻塞，包括 `/api/supabase-config`。

## 3. 端到端链路追踪

### 3.1 场景 1：首次访问 / → proxy → 重定向 /login → 渲染登录页

```
用户访问 /
  ↓
proxy.ts 检查 token
  ↓ 无 token
  ↓ pathname 不在 publicRoutes → redirectToLogin
  ↓
浏览器收到 307 重定向到 /login
  ↓
proxy.ts 检查 /login
  ↓ /login 在 publicRoutes → 放行
  ↓
LoginPage 渲染
  ↓ authLoading=true → 显示 Spinner
  ↓
AuthProvider init()
  ↓ getSupabaseBrowserClientAsync() → waitForConfig(2s)
  ↓   SupabaseConfigProvider fetch /api/supabase-config
  ↓   → getSupabaseCredentials() → loadEnv() → execSync(python3) [可能阻塞 10s]
  ↓   → 返回 { url, anonKey }
  ↓ config 就绪 → supabase.auth.getSession()
  ↓   → 无 session → session=null
  ↓ isLoading=false
  ↓
LoginPage 显示登录表单
```

**瓶颈**：`loadEnv()` 的 `execSync` 可能阻塞 10 秒。`waitForConfig` 超时 2 秒。如果 config 在 2 秒内未就绪，AuthProvider 抛错，5 秒后 isLoading=false，显示登录表单。**最坏情况：7 秒等待**。

### 3.2 场景 2：登录提交 → Supabase auth → set-cookie → 重定向 / → 渲染主页

```
用户提交登录表单
  ↓
supabase.auth.signInWithPassword()
  ↓ 成功
  ↓
fetch("/api/auth/set-cookie", { method: "POST", body: token })
  ↓ 设置 httpOnly cookie
  ↓
document.cookie = "sb-logged-in=true; ..."
  ↓
router.replace("/")
  ↓
proxy.ts 检查 token
  ↓ cookie 中有 sb-access-token → 验证 token
  ↓ supabase.auth.getUser(accessToken) [网络请求，可能慢]
  ↓ 验证通过 → 放行
  ↓
HomePage 渲染
  ↓ authLoading=true → 显示骨架屏
  ↓
AuthProvider init()
  ↓ getSupabaseBrowserClientAsync() → waitForConfig(2s)
  ↓ config 就绪 → supabase.auth.getSession()
  ↓   → 有 session（Supabase SDK 从 localStorage 恢复）
  ↓ setSession / setUser
  ↓ setAuthCookie(token) [未 await，fire-and-forget]
  ↓ fetchProfile(user.id) → fetch /api/profile
  ↓   → authenticateRequest → supabase.auth.getUser [再次网络请求]
  ↓   → db.select() → 查询 profiles 表
  ↓ isLoading=false
  ↓
useFetch("/api/records?limit=50") 触发
  ↓ session.access_token 有值 → fetch /api/records
  ↓ → 返回数据 → loading=false
  ↓
HomePage 渲染数据
```

**瓶颈**：
1. proxy.ts 的 `getUser` 调用（场景 2 第一步）
2. AuthProvider 的 `fetchProfile` 中再次 `getUser`（冗余）
3. `setAuthCookie` 未 await，如果 cookie 未设置完成就跳转，proxy 可能不放行

### 3.3 场景 3：刷新页面 → proxy 验证 token → AuthProvider 初始化 → 加载用户数据

```
用户刷新 /
  ↓
proxy.ts 检查 cookie 中 sb-access-token
  ↓ 有 token → supabase.auth.getUser(accessToken) [网络请求]
  ↓ 验证通过 → 放行
  ↓
HomePage 渲染
  ↓ authLoading=true → 显示骨架屏
  ↓ useFetch 初始化 loading=true
  ↓
AuthProvider init()
  ↓ getSupabaseBrowserClientAsync() → waitForConfig(2s)
  ↓   SupabaseConfigProvider fetch /api/supabase-config
  ↓   → 如果 loadEnv() 首次调用 [阻塞 10s]
  ↓   → waitForConfig 2s 超时 → 抛错
  ↓ catch → finally → isLoading=false
  ↓ session=null（因为 init 失败了）
  ↓
useFetch 检查 session?.access_token → null
  ↓ fetchData 直接 return → loading 永远 true
  ↓
HomePage 永久显示骨架屏 = 一直转圈 ❌
```

**这就是用户遇到的"一直转圈"的核心链路！**

如果 config 加载成功（非首次调用，envLoaded 已缓存）：
```
  ↓ config 就绪 → supabase.auth.getSession()
  ↓   → SDK 从 localStorage 恢复 session
  ↓   → 有 session → setSession / setUser
  ↓   → fetchProfile(user.id) → /api/profile
  ↓     → authenticateRequest → supabase.auth.getUser [网络请求]
  ↓     → db.select() → 查询 profiles
  ↓   → profile 设置完成
  ↓ isLoading=false
  ↓
useFetch 触发 → session 有值 → 正常获取数据
  ↓
HomePage 渲染完成 ✅
```

### 3.4 场景 4：已登录访问 /login → proxy 放行 → AuthProvider 检测 → 重定向 /

```
已登录用户访问 /login
  ↓
proxy.ts 检查 /login → publicRoutes → 放行
  ↓
LoginPage 渲染
  ↓ authLoading=true → 显示 Spinner
  ↓
AuthProvider init()
  ↓ session 有值 → setUser
  ↓ isLoading=false
  ↓
LoginPage useEffect: !authLoading && user → setRedirecting(true)
  ↓ router.replace("/")
  ↓
显示 Spinner（redirecting 状态）
  ↓
跳转到 / → 场景 3
```

**潜在问题**：如果场景 3 也失败（config 加载慢），会进入无限转圈。

## 5. 修复建议

### 5.1 P0 修复（立即执行）

#### 修复 1：`useFetch` loading 死锁（P0-1, P0-2, P0-3）

**文件**：`src/hooks/use-api.ts`

```typescript
const fetchData = useCallback(async () => {
  if (!session?.access_token || !enabled) {
    // 关键修复：session 为空时清除 loading，避免永久转圈
    setState((p) => ({ ...p, loading: false }));
    return;
  }
  // ...
}, [url, session, enabled]);
```

**同样修复**：`src/app/(main)/records/page.tsx` 的 `fetchRecords` 函数开头：
```typescript
const fetchRecords = useCallback(async () => {
  if (!session?.access_token) {
    setLoading(false);  // 关键修复
    return;
  }
  // ...
}, [session?.access_token]);
```

#### 修复 2：添加客户端认证守卫（P0-4）

**文件**：`src/app/(main)/layout.tsx`

```typescript
"use client";
import { useEffect } from "react";
import { useRouter } from "next/navigation";
import { useAuth } from "@/components/auth/auth-provider";
import { AppLayout } from "@/components/layout/app-layout";

export default function MainLayout({ children }: { children: React.ReactNode }) {
  const { user, isLoading } = useAuth();
  const router = useRouter();

  useEffect(() => {
    if (!isLoading && !user) {
      router.replace("/login");
    }
  }, [isLoading, user, router]);

  if (isLoading || !user) {
    return (
      <div className="flex items-center justify-center min-h-screen">
        <div className="w-8 h-8 border-2 border-primary border-t-transparent rounded-full animate-spin" />
      </div>
    );
  }

  return <AppLayout>{children}</AppLayout>;
}
```

#### 修复 3：流式响应超时保护（P0-5, P1-3）

**文件**：`src/app/api/analysis/route.ts` 和 `src/app/api/gateway/route.ts`

```typescript
// 为 fetch 添加 30 秒超时
const response = await fetch(gatewayUrl, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify(payload),
  signal: AbortSignal.timeout(30000),  // 关键修复

// while(true) 循环中添加空闲超时
let lastChunkTime = Date.now();

    controller.error(new Error("Stream timeout"));
    break;
  }
  const { done, value } = await reader.read();
  if (done) break;
  lastChunkTime = Date.now();
  // ...
}
```
