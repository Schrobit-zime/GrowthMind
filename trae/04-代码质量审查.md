## 2. TypeScript 类型安全

### 2.1 类型导入冲突

**文件：** `src/storage/database/supabase-client.ts`

```typescript
// 问题：SupabaseClient 类型名与同名函数冲突
import { createClient, SupabaseClient } from "@supabase/supabase-js";

// 后续定义了同名函数，导致类型标识符被遮蔽
export function SupabaseClient() { ... }
```

**修复建议：** 使用 `type` 关键字显式标记类型导入，或重命名函数避免冲突。

```typescript
import { createClient, type SupabaseClient } from "@supabase/supabase-js";
```

### 2.2 Supabase Client 缺少 Database 泛型

**文件：** `src/storage/database/supabase-client.ts`

`getSupabaseClient()` 返回类型为 `SupabaseClient`，缺少 Database 泛型参数，导致 `.from("records").select("*")` 无法获得正确的类型推断。

**修复建议：**

```typescript
import type { Database } from "./types/database.types";

export function getSupabaseClient(): SupabaseClient<Database> {
  return createClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  );
}
```

### 2.3 Drizzle relations.ts 空文件

**文件：** `src/storage/database/shared/relations.ts`

该文件仅包含导入语句但未定义任何关联关系，表明 Drizzle ORM 的 Schema 定义处于半成品状态。

**修复建议：** 要么补全关系定义并统一使用 Drizzle 作为数据访问层，要么移除 Drizzle 相关代码避免混淆。

### 2.4 缺少类型定义目录

项目中无 `src/types/` 目录，接口定义散落在各组件文件中，导致：

- 类型复用困难
- 跨组件类型一致性无法保证
- 新开发者难以定位共享类型

**修复建议：** 创建 `src/types/` 目录，按职责拆分：

```
src/types/
├── index.ts          # 统一导出
├── database.ts       # 数据库模型类型
├── api.ts            # API 请求/响应类型
└── components.ts     # 组件共享 Props 类型
```

## 3. React/Next.js 最佳实践

### 3.1 根页面未重定向

**文件：** `src/app/page.tsx`

根页面仍显示"应用开发中"默认模板，未重定向到实际应用入口（如 `/dashboard` 或 `/login`）。

**修复建议：**

```typescript
// src/app/page.tsx
import { redirect } from "next/navigation";

export default function RootPage() {
  redirect("/dashboard");
}
```

### 3.2 全部页面标记 'use client'

所有页面组件都使用了 `'use client'` 指令，完全丧失了 Next.js App Router 的 Server Components 优势：

- **SEO 不利：** 搜索引擎无法获取服务端渲染的 HTML
- **首屏性能差：** 需要下载、解析、执行 JS 后才能渲染
- **无法使用 Server Actions：** 数据获取只能通过客户端 API 调用

**修复建议：** 采用 Server/Client 组件分层架构：

```
页面组件（Server Component）
├── 数据获取 ← 在服务端完成
├── 布局结构 ← 服务端渲染
└── 交互组件（Client Component）
    ├── 表单 ← 'use client'
    ├── 图表 ← 'use client'
    └── 按钮/弹窗 ← 'use client'
```

**具体改造方案：**

| 页面 | 当前状态 | 建议方案 |
|------|----------|----------|
| Dashboard | 全客户端 | 页面容器改 Server Component，图表/交互部分抽为 Client Component |
| Record Form | 全客户端 | 表单组件保留 'use client'，页面容器改为 Server Component |
| Supervise | 全客户端 | 列表展示改 Server Component，筛选/操作改 Client Component |

### 3.3 内联 SVG 图表

项目中手写了 100+ 行的内联 SVG 图表代码，而 `package.json` 中已安装 `recharts` 但未使用。

**修复建议：** 使用已安装的 `recharts` 替代手写 SVG：

```typescript
// Before: 100+ 行手写 SVG
<svg viewBox="0 0 400 200">
  <path d="M 0 200 ..." />
  {/* 大量手动坐标计算 */}
</svg>

// After: 使用 recharts
import { AreaChart, Area, XAxis, YAxis, Tooltip } from "recharts";

<AreaChart width={400} height={200} data={data}>
  <Area type="monotone" dataKey="value" stroke="#8884d8" />
  <XAxis dataKey="date" />
  <YAxis />
  <Tooltip />
</AreaChart>
```

### 3.4 Mock 数据硬编码

几乎所有页面组件内部都硬编码了 Mock 数据，导致：

- 组件文件体积膨胀
- 真实数据接入时需要逐一替换
- 无法针对不同环境切换数据源

**修复建议：**

```typescript
// src/mocks/dashboard.ts — 集中管理 Mock 数据
export const mockDashboardData = { ... };

// src/lib/api/dashboard.ts — 数据获取层
export async function getDashboardData() {
  if (process.env.NODE_ENV === "development" && USE_MOCK) {
    return mockDashboardData;
  }
  return fetch("/api/dashboard").then(res => res.json());
}
```

### 3.5 组件文件过大

| 文件 | 行数 | 问题 |
|------|------|------|
| record-form | ~400 行 | 表单逻辑、验证、UI 全部耦合 |
| supervise-rules | ~300 行 | 规则配置、列表、编辑混合 |
| supervise | ~280 行 | 页面布局、数据获取、交互混合 |
| gateway | ~270 行 | 网关配置与展示耦合 |

**修复建议：** 遵循单一职责原则拆分：

```
record-form/
├── index.tsx              # 主组件（组装）
├── record-form-fields.tsx # 表单字段
├── record-form-schema.ts  # 验证规则
├── record-form-actions.ts # 提交逻辑
└── use-record-form.ts     # 自定义 Hook
```

## 4. 样式一致性

### 4.1 shadcn/ui 使用率为零

项目安装了 60+ shadcn/ui 组件，但页面中全部使用手写 HTML + Tailwind CSS 实现，未使用任何 shadcn/ui 组件。

**影响：**

- 浪费了组件库的可访问性（a11y）支持
- 丧失了主题切换能力
- 样式一致性完全依赖人工保证

**修复建议：** 优先替换高频使用的 UI 元素：

| 手写实现 | 替换为 shadcn/ui 组件 |
|----------|----------------------|
| `<button className="...">` | `<Button variant="...">` |
| `<input className="...">` | `<Input />` / `<Textarea />` |
| `<dialog className="...">` | `<Dialog />` / `<Sheet />` |
| `<select className="...">` | `<Select />` |
| `<table className="...">` | `<Table />` |
| 自定义 Toast | `<Sonner />` (已安装) |

### 4.2 样式类名重复

以下样式组合在多个页面中重复出现：

```
bg-surface/40 backdrop-blur-xl border border-border/20 rounded-xl p-4
```

**修复建议：**

1. **使用 `cn()` + 组件抽象：**

```typescript
// src/components/ui/glass-card.tsx
import { cn } from "@/lib/utils";

export function GlassCard({ className, ...props }: React.HTMLAttributes<HTMLDivElement>) {
  return (
    <div
      className={cn("bg-surface/40 backdrop-blur-xl border border-border/20 rounded-xl p-4", className)}
      {...props}
    />
  );
}
```

2. **或使用 Tailwind `@apply` 提取通用样式：**

```css
@utility glass-card {
  @apply bg-surface/40 backdrop-blur-xl border border-border/20 rounded-xl p-4;
}
```

### 4.3 CSS 变量重复定义

**问题：** `@theme inline` 和 `:root` 中的 CSS 变量内容完全相同，存在维护两份同步代码的风险。

**修复建议：** 统一使用一种方案。在 Tailwind CSS v4 中推荐使用 `@theme inline`：

```css
/* 移除 :root 中的重复定义，仅保留 @theme inline */
@theme inline {
  --color-surface: oklch(0.98 0.01 240);
  --color-border: oklch(0.90 0.01 240);
  /* ... */
}
```

## 5. 代码组织

### 5.1 目录结构分析

```
src/
├── app/           # Next.js 路由页面
├── components/    # 通用组件
├── lib/           # 工具函数
├── storage/
│   └── database/
│       ├── supabase-client.ts   # Supabase 客户端
│       └── shared/
│           └── relations.ts     # 空文件，Drizzle 未使用
└── styles/        # 全局样式
```

**问题：**

- 数据库代码在 `src/storage/database/`，但 Drizzle ORM 仅定义了 Schema 未实际使用
- 缺少 `src/services/` 业务逻辑层，页面组件直接操作数据源
- 缺少 `src/types/` 类型定义目录
- 缺少 `src/mocks/` Mock 数据目录

**建议的目标架构：**

```
src/
├── app/              # Next.js 路由（仅布局和页面容器）
├── components/
│   ├── ui/           # shadcn/ui 基础组件
│   └── features/     # 业务功能组件
├── services/         # 业务逻辑层（数据获取、转换）
├── storage/          # 数据访问层（Supabase/Drizzle）
├── types/            # 共享类型定义
├── mocks/            # Mock 数据
├── constants/        # 常量定义
├── hooks/            # 自定义 Hooks
├── lib/              # 工具函数
└── styles/           # 全局样式
```

### 5.2 缺少常量定义

魔法字符串散落在代码各处：

```typescript
// 散落在各文件中的魔法字符串
if (type === "日报") { ... }
if (role === "admin") { ... }
if (status === "active") { ... }
```

**修复建议：** 创建 `src/constants/` 目录集中管理：

```typescript
// src/constants/record.ts
export const RECORD_TYPES = ["日报", "周报", "月报"] as const;
export type RecordType = (typeof RECORD_TYPES)[number];

// src/constants/user.ts
export const USER_ROLES = ["admin", "member", "viewer"] as const;
export const USER_STATUS = ["active", "inactive", "suspended"] as const;
```

### 5.3 错误处理

注：项目实际已包含 `error.tsx` 和 `not-found.tsx`。

但代码中存在以下问题：

- `catch` 块为空，吞掉了所有错误
- 无统一的错误上报机制
- 无加载状态（Loading）组件

**修复建议：**

```typescript
// 1. 统一错误处理
try {
  await fetchData();
} catch (error) {
  console.error("数据获取失败:", error);
  toast.error("操作失败，请稍后重试");
  throw error; // 向上传播
}

// 2. 添加 Loading 状态
// src/app/dashboard/loading.tsx
export default function DashboardLoading() {
  return <DashboardSkeleton />;
}
```

## 7. 正面评价

项目在以下方面做得很好，值得保持：

### 7.1 文件命名规范

统一使用 `kebab-case` 命名文件和目录，符合 Next.js 社区惯例：

```
src/components/record-form.tsx
src/components/supervise-rules.tsx
src/storage/database/supabase-client.ts
```

### 7.2 导入路径别名

正确配置了 `@/` 路径别名，避免了深层相对路径：

```typescript
// ✅ 清晰的别名导入
import { cn } from "@/lib/utils";
import { Button } from "@/components/ui/button";

// ❌ 避免了这种写法
import { cn } from "../../../lib/utils";
```

### 7.3 TypeScript 使用

代码中无 `any` 类型滥用（仅一处 catch 块），表明团队有良好的类型安全意识。

### 7.4 组件导出方式

统一使用 `export default function` 导出组件，与 Next.js App Router 的约定一致：

```typescript
export default function DashboardPage() {
  return <div>...</div>;
}
```

### 7.5 CSS 变量命名

遵循 shadcn/ui 的 CSS 变量命名规范，便于后续主题定制：

```css
--color-surface: ...;
--color-border: ...;
--color-primary: ...;
```

### 7.6 响应式设计

所有页面支持移动端适配，使用 `lg:` 断点进行响应式布局切换，说明团队重视移动端体验。
